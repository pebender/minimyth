General

MiniMyth obtains configuration from the following places:

  • the boot line,
  • the DHCP server,
  • the MiniMyth configuration file,
  • the MiniMyth configuration script and
  • miscellaneous configuration files.

Ideally, the MiniMyth configuration script and the miscellaneous configuration files should not be needed. However, MiniMyth still lacks configuration files for many remote control recievers, remote controls, and LCD and VFD displays. In addition, MiniMyth still lacks the flexibility needed for some X Window System configurations. If you have configuration files for your hardware and setup that you would like included in MiniMyth, then file a bug report containing a list of you hardware including vendor and model, the outputs of the commands 'lspci -mn' and 'lsusb' with all you hardware connected, and the configuration files you want included.

Configuration obtained from the boot line

MiniMyth can obtain configuration from the boot line. The following boot line parameters are of some interest:

  • Linux parameters:
    debug
    Enables kernel debug logging and disables the boot splash screen.
    irqpoll
    Works around IRQ problems with the ASUS A8N-VM CSM motherboard that can cause a a kernel oops when the nvidia kernel driver is loaded, resulting in X not starting correctly.
  • MiniMyth debug variables:
    MM_DEBUG
    Enables MiniMyth debug logging. Valid values are "yes".
  • MiniMyth security variables:
    MM_SECURITY_ENABLED
    See 'MM_SECURITY_ENABLED'.
  • MiniMyth DHCP override variables:
    MM_DHCP_HOSTNAME
    See Hostname.
    MM_DHCP_DOMAIN
    See Domain Name.
    MM_DHCP_DNS_SERVERS
    See Domain Server.
    MM_DHCP_NTP_SERVERS
    See NTP Servers.
    MM_DHCP_LOG_SERVERS
    See Log Server.
    MM_DHCP_TZ
    See TCode.

Unlike in the MiniMyth configuration file where MiniMyth variable values should be single quoted ('), MiniMyth variable values on the boot line must either be unquoted or double quoted (").

Configuration obtained from the DHCP server

MiniMyth can obtain configuration from the DHCP server.

Hostname

The MiniMyth frontend's hostname. MiniMyth requires a hostname for storing/retrieving the MythTV frontend settings in/from the MythTV's database. Therefore, if MiniMyth does not obtain a hostname, then MiniMyth will abort its boot process and output an error.

The DHCP option is 12 (0x0c), which is supported by ISC BIND. Therefore, if your DHCP server is ISC BIND, then you should enable the 'use-host-decl-names' option and set the 'host' and 'hardware' options in your dhcpd.conf file, and not set the corresponding DHCP override variable either on the boot line ('MM_DHCP_HOSTNAME') or in the MiniMyth configuration file ('MM_HOSTNAME').

In addition to using the hostname when making MythTV database accesses, MiniMyth can use the hostname when retrieving host specific configuration files. For this to work for the 'minimyth.conf' file, the hostname must be set prior to retrieving the 'minimyth.conf' file. Setting parameters on the boot line makes them available at the beginning of the boot process. Therefore, if your DHCP server does not provide the hostname, then you should consider using the DHCP override variable on the boot line ('MM_DHCP_HOSTNAME') and not the DHCP override variable in the or in the MiniMyth configuration file ('MM_HOSTNAME').

Domain Name

The MiniMyth frontend's domain. MiniMyth does not require a domain name, but it can be useful when using DNS to resolve local network hostnames to IP addresses.

The DHCP option is 15 (0x0f), which is supported by ISC BIND. Therefore, if your DHCP server is ISC BIND, then you should set the 'domain-name' option in your dhcpd.conf file, and not set the corresponding DHCP override variable either on the boot line ('MM_DHCP_DOMAIN') or in the MiniMyth configuration file ('MM_DOMAIN').

There is a special value of '.'. After override processing, if the value is '.', then it is treated as though no value was provided.

Domain Server

A space separated list of Domain Name System (DNS) server IP addresses that the MiniMyth frontend will use to resolve DNS names. MiniMyth needs a DNS server to resolve Internet host name to Internet Protocol addresses. Therefore, if MiniMyth does not obtain a DNS server address, then MiniMyth will abort its boot process and output an error message.

The DHCP option is 6 (0x06), which is supported by ISC BIND. Therefore, if your DHCP server is ISC BIND, then you should set the 'domain-name-servers' option in your dhcpd.conf file, and not set the corresponding DHCP override variable either on the boot line ('MM_DHCP_DNS_SERVERS') or in the MiniMyth configuration file ('MM_DNS_SERVERS').

NTP Servers

A space separated list of Network Time Protocol (NTP) server DNS names or IP addresses that the MiniMyth frontend will use to synchronize its clock. MiniMyth's clock needs to be synchronized to the MythTV backend's clock or fastforward, rewind and commercial skip will fail. Therefore, if MiniMyth does not obtain an NTP server address, then MiniMyth will abort its boot process and output an error message.

The DHCP option is 42 (0x2a), which is supported by ISC BIND. Therefore, if your DHCP server is ISC BIND, then you should set the 'ntp-servers' option in your dhcpd.conf file, and not set the corresponding DHCP override variable either on the boot line ('MM_DHCP_NTP_SERVERS') or in the MiniMyth configuration file ('MM_NTP_SERVERS').

Log Server

A space separated list of syslog server DNS names or IP addresses to which MiniMyth will log. The value can be a list of log servers or 'none'. MiniMyth will log to the first log server in the list. If MiniMyth does not obtain a log server address, then MiniMyth will log locally.

The DHCP option is 7 (0x07), which is supported by ISC BIND. Therefore, if your DHCP server is ISC BIND, then you should set the 'log-servers' option in your dhcpd.conf file, and not set the corresponding DHCP override variable either on the boot line ('MM_DHCP_LOG_SERVERS') or in the MiniMyth configuration file ('MM_LOG_SERVERS').

There is a special value of '.'. After override processing, if the value is '.', then it is treated as though no value was provided.

TCode

The tz database (a.k.a. the zoneinfo database) name of your MiniMyth frontend's time zone. MiniMyth's time zone needs to be correct or fastforward, rewind and commercial skip will fail. Therefore, if MiniMyth does not obtain time zone, then MiniMyth will abort its boot process and output an error message.

The DHCP option is 101 (0x65), which is not supported by ISC BIND. Therefore, if your DHCP server is ISC BIND, then you should define and set the 'tcode' option in your dhcpd.conf file, and not set the corresponding DHCP override variable either on the boot line ('MM_DHCP_TZ') or in the MiniMyth configuration file ('MM_TZ').

As mentioned above, if you want to use the 'tcode' option in ISC BIND, then you will need to define it in your dhcpd.conf file. As defined in RFC 4833, the 'tcode' option is a string identified by the tag 101 (0x65). As explained in the dhcp-options man page, this new option is defined by adding the following line to dhcpd.conf: 

                option tcode code 101 = test;

After that, you can use 'tcode' as you would any other option, with the option value being the tz database name for your time zone. For example: 

                option tcode "America/Los_Angeles";
Location of configuration files

MiniMyth looks for configuration files and scripts in the MiniMyth configuration directories. There are two MiniMyth configuration directories: the read-only configuration directory and the read-write configuration directory.

The read-only MiniMyth configuration directory is named 'conf' and is required. This directory (actually a sub-directory of this directory) is where you would put configuration files that you create. This directory may be read-only.

The read-write MiniMyth configuration directory is named 'conf-rw' and is optional. This directory is where MiniMyth put files that it creates. For example, when MiniMyth saves game configurations it saves them in this directory. This directory is only required when you are using a MiniMyth feature that saves files across reboot. This directory must be read-write. In addition, if this directory is on a TFTP server, then you need to be sure that the TFTP server is configured to allow file creation not just file upload.

MiniMyth looks for the MiniMyth configuration directories in the MiniMyth boot directory. MiniMyth 'guesses' the location of the boot directory as follows. If the file system contains the directory '/minimyth', then MiniMyth assumes that '/minimyth' is the boot directory. Otherwise, MiniMyth assumes the boot directory is the directory on the TFTP server that contains the boot loader.

In practice, this means that if you network boot MiniMyth then you put the MiniMyth configuration directories in the same directory on the TFTP server as the network boot loader (called 'pxelinux.0' when using PXELinux), and that if you local boot MiniMyth, then you put the MiniMyth configuration directories at the root of the partition labeled 'minimyth', which MiniMyth will automatically mount at the mount point '/minimyth'.

Once MiniMyth has 'guessed' the boot directory (lets call it '{boot}'), then MiniMyth looks for configuration files first as the file '{boot}/conf-rw/{hostname}+{file}' second as the file '{boot}/conf/{hostname}/{file}' and third as the file '{boot}/conf/default/{file}', where '{hostname}' is the hostname of the MiniMyth frontend and '{file}' is the name of the configuration file.

Configuration obtained from the MiniMyth configuration file

The MiniMyth configuration file is called 'minimyth.conf'. The MiniMyth configuration file is required. You can find information on the configuration that can be obtained from MiniMyth configuration file in the minimyth.conf description.

Configuration obtained from the MiniMyth configuration script

The MiniMyth configuration script is called 'minimyth.script'. The MiniMyth configuration script is optional. The MiniMyth configuration script is a shell script that is executed by MiniMyth after MiniMyth has retrieved and processed the MiniMyth configuration file and configured name (DNS) and logging (syslog) services. You can use the MiniMyth configuration script to do things such as change the startup or shutdown scripts. Whether or not 'minimyth.script' is retrieved depends on 'MM_MINIMYTH_FETCH_MINIMYTH_SCRIPT'.

Configuration obtained from miscellaneous configuration files

MiniMyth can override the following configuration files:

ssh_host_rsa_key

An SSH authorized keys file. This file contains the public keys of users that are allowed to connect to the SSH server as the user 'root' using public key authentication. The format of these files must be in the format of an OpenSSH authorized keys file. Whether or not 'authorized_keys' is retrieved depends on 'MM_SSH_SERVER_FETCH_AUTHORIZED_KEYS'.

ca_bundle_crt

The bundle digital certificates for trusted certificate authorities. Whether or not 'ca-bundle.crt' is retrieved depends on 'MM_SECURITY_FETCH_CA_BUNDLE_CRT'.

crontab

The crontab file for user root. Whether or not 'crontab' is retrieved depends on 'MM_CRON_FETCH_CRONTAB'.

id_rsa

An SSH private RSA identity file. This file is the identity of SSH user ('MM_SSH_REMOTE_USER') when communicating with the remote command execution host ('MM_SSH_REMOTE_HOST'). The format of these files must be in the format of an OpenSSH private RSA key as generated by ssh-keygen. Whether or not 'id_rsa_remote' is retrieved depends on 'MM_SSH_REMOTE_FETCH_ID_RSA'.

WARNING: Because access to the TFTP server storing this file is not secure, this SSH RSA private identity file should not be trusted. You should assume that because it can be compromised that it will be compromised. Therefore, do not allow anything identifying itself using this identity file to do anything that could be harmful.

LCDd.conf

The LCDproc daemon ('LCDd') configuration file. MiniMyth will use this file as the '/etc/LCDd.conf' file in the MiniMyth file system. Whether or not 'LCDd.conf' is retrieved depends on 'MM_LCDPROC_FETCH_LCDD_CONF'.

lircd.conf

The LIRC daemon ('lircd') configuration file. MiniMyth will use this file as the '/etc/lircd.conf' file in the MiniMyth file system. Whether or not 'lircd.conf' is retrieved depends on 'MM_LIRC_FETCH_LIRCD_CONF'.

lircrc

The LIRC configuration file. MiniMyth will use this file as the '/etc/lircrc' file in the MiniMyth file system. Whether or not 'lircrc' is retrieved depends on 'MM_LIRC_FETCH_LIRCRC'.

xinitrc

The X Window System (Xorg) startup file. MiniMyth will use this file as the '/etc/X11/xinit/xinitrc' file in the MiniMyth file system. Whether or not 'xinitrc' is retrieved depends on 'MM_X_FETCH_XINITRC'.

ssh_host_rsa_key

An SSH private RSA identity file. This file is the host key of the SSH server. The format of these files must be in the format of an OpenSSH private RSA key as generated by ssh-keygen. Whether or not 'ssh_host_rsa_key' is retrieved depends on 'MM_SSH_SERVER_FETCH_SSH_HOST_RSA_KEY'.

WARNING: Because access to the TFTP server storing this file is not secure, this SSH RSA private identity file should not be trusted. You should assume that because it can be compromised that it will be compromised. Therefore, do not allow anything identifying itself using this identity file to do anything that could be harmful.

xorg.conf

The X Window System (Xorg) configuration file. MiniMyth will use this file as the '/etc/X11/xorg.conf' file in the MiniMyth file system. Whether or not 'xorg.conf' is retrieved depends on 'MM_X_FETCH_XORG_CONF'.

If you want to use an LCD or VFD in your MiniMyth frontend, and you do not like the 'LCDd.conf' file that MiniMyth uses for your LCD or VFD, then you can provide a different 'LCDd.conf' file.

If you want to control your MiniMyth frontend with a remote control, and either your remote control receiver or remote control is not one of the built in remote control receivers and remote controls, then you will need to provide the appropriate 'lircd.conf' and 'lircrc' files.

If you cannot configure the X Window System to your liking using the X configuration variables in the MiniMyth configuration file, then you can provide different 'xinitrc' and/or 'xorg.conf' files.

If you want to override any other configuration files, then you will need to use the MiniMyth configuration script.
